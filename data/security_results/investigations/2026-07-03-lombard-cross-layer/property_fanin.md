## Lead hypotheses (ranked seeds)

| # | Hypothesis | Primary Property | Subsystem / Surface |
|---|---|---|---|
| H1 | Cross-layer replay between EVM Mailbox.delivered / usedPayloads and Solana mailbox `MessageV1Info.status` and `bridge.message_handled` PDA. Mailbox handle_message sets `Handled` BEFORE the recipient CPI returns. If the recipient CPI fails the message is still marked Handled — partial-handle desync across `message_handled` PDA initialized inside `gmp_receive` and `Mailbox.message_info`. | `PROP-XR-001` | lombard_token_pool.release_or_mint → mailbox.handle_message → bridge.gmp_receive |
| H2 | `mailbox.handle_message`'s `destination_caller` check compares the 32-byte handler key against the message's `destination_caller`. In `lombard_token_pool.release_or_mint_tokens`, the handler is the token-pool `state` PDA, while `pool_signer` is only a remaining account for bridge CPI. Misconfiguring `destination_caller` to `pool_signer` should fail. | `PROP-XR-002` | mailbox.handle_message |
| H3 | Bascule GMP `validate_mint` ↔ `report_mint` ↔ Solana `bridge.message_handled` ↔ `mailbox.message_info.status` boundary. In `gmp_receive`, the tapis assertion that `computed_payload_hash == payload_hash` is inside the mailbox PDA scope (seeds `MESSAGE_SEED, payload_hash`) — if the same payload_hash is delivered twice with different bodies, does Anchor's `init` constraint on `message_handled` block the second? | `PROP-XR-003` | bridge.gmp_receive + mailbox.handle_message |
| H4 | Consortium `_checkProof` / quorum / sig alignment — `post_session_signatures` indexes into `current_validators[*index as usize]`. If the `index` provided is within the *current* validator set length but the validator at that index has been rotated mid-session, the `signed` boolean doesn't reset (Array stays the same size because each user has their own `Session` PDA), but the validator pubkey at that index changes → spurious quorum auth possible. | `PROP-CR-007` (extension of PROP-CONS-002) | consortium.post_session_signatures + update_valset interleaving |
| H5 | LombardTokenPool handler's `bridge_token_authority` is a separate PDA. Signature composition bug: `bridge.deposit` is invoked via `pool_signer` PDA via `state` + `token_pool_signer_seeds`. If the bytes-pattern of token_pool_signer equals a mis-deriveable account, a wrong signer could authenticate. **Subsumed by Anchor seed constraint; recheck.** | `PROP-TP-001` (close recheck) | lombard_token_pool.lock_or_burn_tokens |
| H6 | Storage / boundary desyncs: `consortium.Session.space = 8 + 4 + config.current_validators.len() + 8` and `Session.signed = vec![false; config.current_validators.len()]`. If `config.current_validators.len()` shrinks (`close_session_for_epoch`), previously-counted `signed` weights could over-quorum at any retry. (`close_session_for_epoch` only closes a *user's session payload* by re-quorum check, never shrinks `current_validators`.) | `PROP-SD-001` | consortium storage layout + close_session_for_epoch |
